1. Project Overview & Problem Statement

This project is a modular Enterprise Resource Planning (ERP) application under active development. It uses a React 19 / Vite single-page application (SPA) for the operational interface and a TypeScript Express API backed by PostgreSQL through Prisma ORM. The foundation currently concentrates on Identity and Access Management (IAM): user creation, role assignment, a permissions matrix, login, and protected client routes.

An ERP becomes difficult to evolve when each business domain owns isolated data and authorization rules. The design addresses that problem by establishing a shared identity layer and a consistent API/data boundary before domain modules are added. This supports:

Consistent Authorization

Permissions are modeled as data rather than scattered as hard-coded UI conditions across separate views.

Composable Modules

Future ERP modules can be represented as permission records and implemented as independent API layers without changing the IAM core.

Relational Integrity

PostgreSQL relationships prevent orphaned records; Prisma provides a strictly-typed client layer over them.

Operational Scale

A separate SPA and API tier allow each layer to be deployed and scaled independently behind Nginx proxy routing.

Architectural Note on Multi-Tenancy

The present schema is single-organization. Multi-tenancy is a planned extension. A robust future iteration will introduce an Organization schema and require tenant boundaries on all domain queries and unique constraints.

3. Database Design & Schema Modeling

Prisma splits schemas into logical components (base.prisma and iam.prisma) which compile down to native versioned tables. Service methods use strictly-typed hooks to query data.

Model Name Purpose and Relationships
User Unique indexed userEmail, bcrypt password hash, and audit timestamps. Many-to-many relationship with roles.
Role A named authority grouping connecting users and unique application permission rules via RolePermission.
Permission Master catalog of module boundaries, identified by a unique key matching features (e.g., students).
RolePermission Explicit compound primary key matrix holding dynamic contextual access decisions: canRead, canAdd, canEdit, canDelete.

4. Security & Authentication Architecture

Granular Authorization Metrics

Server-side computations dynamically process access metrics across intersecting data layers via an allow-if-any-role-allows architectural standard.

Security Hardening Items Checklist

  • Mount core validation middleware uniformly across all non-public endpoints.
  • Enforce rigorous server-side authorization evaluation instead of purely relying on visual UI layer guards.
  • Transition tokens from browser local storage over to cryptographic, secure HTTP-only cookies.

CI/CD Engine Implementations

web.erp Pipeline Target: /var/www/web.erp

Enforces strict environment secret validations before configuring a Node 20 build image. Compiles optimized production static sets using clean tree-shaking parameters, transmits artifacts over port 22 using Appleboy SCP actions, resolves web server directory ownership flags to www-data, and triggers a clean, zero-downtime Nginx configuration hot reload.

api.erp Pipeline Target: /root/api.erp

Spins up an Ubuntu Virtual Environment running Node 22. Injects strict database parameters to generate fully optimized, type-safe Prisma binary client sets before initiating build steps. Transmits files while excluding developmental telemetry dependencies via --omit=dev flags, safely triggers localized environment NVM setups, and hot-starts the backend runtime process loop via PM2 script bindings.
